LOCAL JARVIS-STYLE JOB APPLICATION AGENT - BUILD PROMPT
==========================================================

----------------------------------------------------------------------
1. ROLE
----------------------------------------------------------------------

You are a senior AI systems architect, Windows desktop automation engineer, local-LLM engineer, and privacy-first product designer.

Design a fully local Windows desktop AI agent that runs on my own laptop and helps me apply for jobs. The agent must work without OpenAI API, Anthropic API, Gemini API, paid API keys, hosted LLM calls, or any cloud inference dependency.

The target machine is an Alienware mR2 laptop with an RTX 4090 GPU. Assume I am willing to use local GPU/CPU power and download open-source models, but I do not want recurring cost. The system must be built primarily for Windows.

Think of the product as a local Jarvis-style computer assistant:
- It can see the screen.
- It can move the mouse.
- It can click.
- It can type.
- It can read local files.
- It can search/index local repositories, folders, resumes, cover letters, trackers, notes, and documents.
- It can ask me questions when it is unsure.
- It can learn from my actions when learning mode is enabled.
- It can remember user-approved answers and reuse them later.
- It can use the internet only after user permission.
- It must not depend on any external LLM API key.

Do not give a vague concept. Produce a concrete build specification, architecture, component list, storage plan, permissions model, and implementation roadmap.

----------------------------------------------------------------------
2. CORE PRODUCT VISION
----------------------------------------------------------------------

Build a privacy-first local desktop assistant for job applications.

The agent should behave like an intelligent assistant sitting beside me:
- When I work, it can watch the screen if I enable screen mode.
- When I take control, it should observe and learn reusable steps if learning mode is enabled.
- When it works, it should minimise itself to the Windows tray.
- It should interrupt only when required using small notification-style prompts.
- I should be able to talk to it using voice.
- I should also be able to chat with it in a modern futuristic desktop UI and via the small notification chat when it is unsure.
- It should maintain memory across days instead of losing context.
- It should manage local files safely, including resumes, cover letters, trackers, downloaded documents, and job notes.

The agent should not blindly submit final applications. Final submission must require explicit user approval unless I later create a trusted rule for a specific site/workflow.

----------------------------------------------------------------------
3. STRICT CONSTRAINTS
----------------------------------------------------------------------

A. Local-only intelligence:
- No paid LLM API.
- No cloud inference.
- No mandatory SaaS dependency.
- Use local models through Ollama, llama.cpp, LM Studio local server, Transformers, UI-TARS, or similar open-source/local runtimes.
- The internet may be used only through the browser and only after user permission.

B. Windows-first:
- Primary operating system is Windows.
- Use Windows-compatible automation libraries.
- Use tray support and desktop notifications.
- Support Chrome automation because the main workflow uses Chrome.

C. Vision-first operation:
- The agent should be able to understand screen state using screenshots and vision-language models. The screenshots should automatically be deleted to clear up space after the agent has understood.
- It should not rely only on brittle UI field detection.
- It may use DOM automation, Playwright, Chrome DevTools Protocol, accessibility tree, or UI Automation API as reliability helpers, but the core design must include screen vision.

D. Human-in-the-loop:
- Ask the user before:
  - submitting a job application,
  - answering unknown eligibility/sponsorship/security questions,
  - modifying original resume files,
  - sending personal information,
  - accessing new folders,
  - using the internet,
  - solving CAPTCHA or anything that looks like a bot-check.
- Never bypass CAPTCHA, paywalls, anti-bot systems, or website security.
- Respect website terms and avoid spam application behaviour.

E. Data safety:
- Make backups before editing resume files, cover letters, or Excel trackers.
- Keep logs locally.
- Store secrets locally and encrypted.
- Allow memory export/delete.
- Use file allowlists and deny sensitive folders by default unless explicitly approved.

F. Low-token / high-impact:
- Use compressed state summaries.
- Use retrieval instead of dumping all memory into every prompt.
- For screenshots, crop relevant regions before sending to the local vision model.
- Store procedural memory as reusable steps, not huge raw histories.
- Store job application outcomes in structured form.
- Prefer deterministic tools for file operations instead of asking the LLM to rewrite entire files.

5. JOB APPLICATION WORKFLOW TO SUPPORT
----------------------------------------------------------------------

The target workflow is:

1. User opens Chrome and logs into required websites manually.
   Websites may include LinkedIn, Glassdoor, company career sites, Workday, Greenhouse, Lever, SmartRecruiters, Ashby, Indeed, or other job boards.

2. User opens 8 ChatGPT project tabs manually in Chrome.
   These tabs are already logged in and ready.
   The agent must not use OpenAI API. It may interact with ChatGPT web UI only through browser/screen automation if the user explicitly permits.

3. User starts the local agent and enters job search instructions in the UI chat.
   Example:
   "Search Senior Java Backend Engineer jobs in London, avoid Easy Apply, prefer sponsor-friendly companies."

4. Agent searches job websites from Chrome.
   It should:
   - search by title, location, keywords,
   - ignore Easy Apply jobs when requested,
   - identify jobs with external application routes,
   - open job descriptions,
   - capture job title, company, location, job link, posting date if visible, and JD text.

5. Agent collects 8 suitable jobs.
   For each job:
   - open one of the 8 ChatGPT project tabs,
   - paste a structured prompt containing the JD and job link,
   - wait for ChatGPT to score the job and generate cover letter/resume changes,
   - extract score, verdict, missing keywords, resume changes, salary if present, and cover letter download link/file.

6. Agent downloads cover letters generated by ChatGPT.
   It should:
   - save each cover letter into a structured local folder,
   - rename files consistently:
     CompanyName_Role_Cover_Letter.docx
   - record the file path in the tracker.

7. Agent updates the application tracker Excel file.
   The tracker path will be provided by the user.
   It should store:
   - date,
   - company,
   - role,
   - location,
   - job URL,
   - job board/source,
   - score,
   - verdict,
   - cover letter path,
   - resume version path,
   - application status,
   - notes,
   - questions asked,
   - user answers learned.

8. Agent opens the resume file provided by the user.
   It should:
   - create a backup copy,
   - apply only the resume changes proposed by ChatGPT,
   - preserve formatting where possible,
   - create a job-specific resume copy instead of overwriting the master resume,
   - show a diff/preview if possible,
   - ask user approval before final save.

9. Agent returns to the job site and applies.
   It should:
   - fill forms,
   - upload tailored resume,
   - upload generated cover letter,
   - answer known questions from memory,
   - ask the user when unsure,
   - save new answers to memory only after approval,
   - stop before final submit and ask for confirmation.

10. Agent records the final status.
   Examples:
   - Applied
   - User Review Needed
   - Blocked by Login
   - Blocked by CAPTCHA
   - Missing Answer
   - Site Error
   - Not Suitable
   - Duplicate Application

6. MEMORY REQUIREMENTS
----------------------------------------------------------------------

The agent must have persistent memory across sessions.

Memory types:

A. Profile memory:
- user name,
- resume paths,
- cover letter template paths,
- job tracker path,
- work authorisation preferences,
- sponsorship wording,
- preferred job titles,
- target locations,
- salary expectations,
- safe answers for repeated forms,
- unsafe questions that always require confirmation.

B. Application memory:
- every job found,
- every job applied,
- company,
- role,
- URL,
- score,
- date,
- status,
- cover letter path,
- resume path,
- form answers used,
- problems encountered.

C. Procedural memory:
- how to operate LinkedIn search,
- how to operate Glassdoor search,
- how to fill Workday,
- how to fill Greenhouse,
- how to fill Lever,
- how to use 8 ChatGPT project tabs,
- how to update the tracker,
- how to name files.

D. Document memory:
- indexed local folders,
- resume content,
- cover letter templates,
- GitHub repositories,
- notes,
- previous answers,
- job tracker records.

E. Conversation memory:
- summaries of previous conversations with the agent,
- decisions made,
- approved/rejected answers.

Recommended memory architecture:
- SQLite for structured memory and audit logs.
- SQLite FTS5 for local text search.
- LanceDB or Chroma for vector search over files, notes, repos, JDs, previous answers, and documents.
- Local embeddings model such as bge-small/en, nomic-embed-text, or another open embedding model through Ollama/Transformers.
- Store raw files separately; store embeddings and metadata in local index.
- Add a memory manager that decides what to save, what to summarise, and what to forget.

7. LOCAL MODEL REQUIREMENTS
----------------------------------------------------------------------

The design should recommend local models based on RTX 4090 laptop hardware.

A. Text reasoning model:
Use one of:
- Qwen3 / Qwen2.5 family instruct model,
- DeepSeek-R1 distilled model,
- Llama-family instruct model,
- Mistral-family instruct model,
- Gemma-family instruct model.

Recommended starting point:
- 14B or 32B quantised instruct model for reasoning if VRAM allows.
- 7B/8B model for fast background tasks.
- Separate coder model if needed for file parsing, scripts, or repository Q&A.

B. Vision-language model:
Use one of:
- Qwen2.5-VL 7B/32B,
- UI-TARS 7B for GUI action prediction,
- LLaVA / Llama vision model if easier to run locally.

The vision model should:
- read screenshots,
- identify buttons/fields,
- describe current screen state,
- infer next UI action,
- verify whether a click/type action succeeded.

C. Speech-to-text:
Use:
- faster-whisper with CUDA, or
- whisper.cpp.

D. Text-to-speech:
Use:
- Piper,
- another fully local open-source TTS engine.

E. Embeddings:
Use:
- nomic-embed-text,
- bge-small / bge-base,
- other local embedding model.

F. Model runtime:
Use one of:
- Ollama for easiest local model management,
- llama.cpp for high-performance local inference,
- LM Studio local server for GUI-driven local model management,
- Transformers/PyTorch for vision models needing custom pipelines.

The final design must include fallback model sizes because laptop RTX 4090 VRAM may be lower than desktop RTX 4090 VRAM. Detect actual VRAM at startup.

11. REQUIRED INTERNAL WORKFLOW DESIGN
----------------------------------------------------------------------

Implement the agent as a state machine, not as a single uncontrolled autonomous loop.

Recommended states:

1. IDLE
- Waiting for user command.

2. PREPARE
- Check Chrome is open.
- Check user is logged in.
- Check 8 ChatGPT tabs are available if the user enables ChatGPT-web workflow.
- Check tracker path.
- Check resume path.
- Check download folder.
- Check internet permission.

3. SEARCH_JOBS
- Search LinkedIn/Glassdoor/company sites.
- Apply user filters.
- Skip Easy Apply if requested.
- Collect candidate jobs.

4. EXTRACT_JD
- Open job page.
- Capture title, company, link, JD text, location, posting date.
- Save local JD snapshot.

5. SEND_TO_CHATGPT_TAB
- Switch to a ChatGPT project tab.
- Paste structured prompt.
- Wait for answer.
- Parse score/verdict/changes.
- Download cover letter if generated.

6. UPDATE_TRACKER
- Open or directly edit Excel tracker.
- Add structured row.
- Save safely.

7. TAILOR_RESUME
- Create backup.
- Apply proposed changes to job-specific resume copy.
- Ask user for review if changes are non-trivial.

8. APPLY_ON_SITE
- Return to job site.
- Fill application form.
- Upload resume and cover letter.
- Ask user for unknown answers.

9. FINAL_REVIEW
- Show summary:
  - company,
  - role,
  - files uploaded,
  - key answers,
  - score,
  - tracker status.
- Ask final submit confirmation.

10. SUBMIT_OR_SAVE
- Submit only after user approval.
- Otherwise save as "User Review Needed".

11. MEMORY_UPDATE
- Save safe learned answers.
- Save procedural notes.
- Save application status.

12. DONE
- Show completion summary.

13. ERROR_RECOVERY
- Screenshot error.
- Explain what happened.
- Offer retry, skip, or hand control to user.

Job URL:
[URL]

Company:
[COMPANY]

Role:
[ROLE]

Location:
[LOCATION]

Job Description:
[JD TEXT]

Required output:
1. Score out of 10 using my job-matching framework.
2. Short apply/skip verdict.
3. Salary range if available.
4. JD keywords missing from my resume.
5. Exact resume lines to update, with minimal truthful changes only.

21. FINAL NOTE
----------------------------------------------------------------------

The goal is not to build a toy chatbot.

The goal is a local Windows desktop agent that can:
- reason locally,
- see the screen,
- use Chrome,
- manage files,
- talk by voice,
- remember user-approved facts,
- learn workflows,
- apply for jobs safely,
- and keep the user in control.

Start with a reliable MVP. Add autonomy gradually. Safety, memory, and local-only operation are mandatory from day one.
